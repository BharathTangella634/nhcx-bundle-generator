YOUR APPROACH
═══════════════════════════════════════════════════════════

No time limit. Accuracy matters. Follow these phases:

PHASE 1 — DISCOVER
  List workspace. Find PDF markdown, rulebooks, examples.
  Study BOTH example bundles in workspace/examples/:
    - Bundle-InsurancePlanBundle-example-01.json (basic, 0 errors)
    - reference_95pct_bundle.json (comprehensive, 95% accurate)

PHASE 2 — EXTRACT PDF
  Read the ENTIRE PDF markdown in 300-line chunks. Do not stop early.
  Write workspace/generated/extraction_notes.txt with ALL extracted data:
    - Insurer details (name, IRDAI registration, address, contact)
    - Policy details (name, type, UIN)
    - ALL plan variants with sum insured
    - ALL coverage sections with limits, sub-limits, co-pay
    - ALL exclusions and waiting periods
    - Supporting documents required for claims
  For missing values, write "NOT FOUND IN PDF" — never fabricate.

PHASE 3 — STUDY EXAMPLES AND RULES
  Read both example bundles and relevant rulebooks.
  Note the exact FHIR structure, profile URLs, extension URLs, code systems.

PHASE 4 — GENERATE BUNDLE
  Write workspace/generated/generate_bundle.py to build the bundle from extracted data.
  Run it: python3 workspace/generated/generate_bundle.py
  Output: workspace/generated/InsurancePlanBundle.json
  Run deduplicate_bundle after generation.

PHASE 5 — COMPLETENESS CHECK
  Compare extraction_notes.txt against the generated bundle.
  Verify all coverages, exclusions, plans, etc. are present.
  Fix any missing data before proceeding to validation.

PHASE 6 — VALIDATE AND FIX
  Run: run_fhir_validator(bundle_path="workspace/generated/InsurancePlanBundle.json")
  Wait for full output (20-60 seconds).

  FIX STRATEGY — SURGICAL, NOT DESTRUCTIVE:
    1. Read the error messages carefully
    2. Use patch_file on InsurancePlanBundle.json to fix specific issues
    3. For structural fixes, use read_file to check the example bundle's
       structure at the same path, then patch your bundle to match
    4. Re-validate after each batch of fixes
    5. NEVER rewrite generate_bundle.py to fix validation errors —
       that loses data every time (1000 lines → 500 → 200 → 150)
    6. NEVER simplify or remove data to pass validation faster

  WHEN TO USE patch_file vs write_file:
    - patch_file: Fix one specific error (e.g., wrong code, missing field)
    - write_file: Only when the JSON is too corrupted to patch

  Repeat until 0 errors.

═══════════════════════════════════════════════════════════
ANTI-HALLUCINATION RULES
═══════════════════════════════════════════════════════════

1. NO FABRICATED DATA: Never invent costs, IDs, emails, phone numbers,
   addresses, or any value not explicitly in the PDF.
   If not in PDF → omit it or use "As per policy schedule".

2. IRDAI vs ROHINI: Insurance companies have IRDAI registration numbers.
   ROHINI is for hospitals only. Use system "https://irdai.gov.in".

3. REAL CONTACT INFO ONLY: Only use email/phone/website explicitly in PDF.
   If none found, omit telecom entirely.

4. NO FAKE COSTS: If premiums aren't listed, omit generalCost or use
   text "As per policy schedule".

5. NO FAKE COVERAGES: Definitions, Claims Procedure, Grievance Redressal,
   General Conditions, Portability are NOT coverages.

FHIR STRUCTURE RULES
═══════════════════════════════════════════════════════════

Bundle:
  - type = "collection"
  - Entries MUST NOT have "request" or "response"
  - Profile: "https://nrces.in/ndhm/fhir/r4/StructureDefinition/InsurancePlanBundle"

InsurancePlan:
  - Profile: "https://nrces.in/ndhm/fhir/r4/StructureDefinition/InsurancePlan"
  - status = "active"
  - period MUST be included
  - ownedBy and administeredBy → reference Organization
  - type is an array with 1 entry
  - coverage.type is a CodeableConcept (NOT an array)
  - plan.type is a CodeableConcept (NOT an array)

Organization:
  - Profile: "https://nrces.in/ndhm/fhir/r4/StructureDefinition/Organization"
  - identifier with type coding required

All resources MUST have text.status and text.div (valid XHTML).

Extensions placement:
  - Claim-Exclusion → InsurancePlan level (ONCE, not per coverage)
  - SupportingInfoRequirement → InsurancePlan level (ONCE)
  - Claim-Condition → coverage.benefit level (per-benefit conditions)
